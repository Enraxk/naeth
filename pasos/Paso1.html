§00tl;dr

Tres titulares en cinco frases #

Si solo lees esto, lleva tres cosas: quién es el mejor ciudadano, quién es el más potente pero menos cooperativo, y cuál es el principal riesgo de adoptar uno como base.

mejor ciudadano[ok]

Basic Memory

Su storage es Markdown en disco. Cualquier sistema externo escribe/lee los mismos archivos sin pedir permiso; watchfiles reindexa automáticamente. Triple superficie (MCP + REST + CLI), tool annotations bien declaradas. Único bemol: licencia AGPL-3.0 — fricción si Naeth se monetiza como SaaS.

más potente, menos cooperativo[info]

Hindsight

91.4% LongMemEval (Gemini-3), 83.6% con OSS-20B local. El valor está en el stack TEMPR + RRF + cross-encoder funcionando como sistema cerrado. Viable como caja-negra federada detrás de router, no como librería de componentes sueltos. No acepta embeddings precomputados.

mayor riesgo de adopción[risk]

mcp-memory-service

Bus factor 1 (single maintainer doobidoo, 2298 commits vs siguiente 36). Cadencia 1 release cada 3 días → mantener fork tight es caro. Mejor integrar vía REST/MCP estable que forkear. La inconsistencia README "24 tools" vs código "~18 + aliases" da pista del estilo.

decisión

Ningún sistema del top 5 cumple los 6 ejes a la vez. El 40/40 hipotético solo es alcanzable componiendo capas especializadas — esa es la tesis a investigar en el Paso 2, no a asumir aquí.

aSuperficie de integración

~18 MCP tools modernas en server_impl.py (no 24 como dice el README — el resto son aliases legacy en compat.py). REST FastAPI con 76 endpoints, OAuth 2.1 con Dynamic Client Registration, header X-Agent-ID. Dashboard React con 8 pestañas y SSE en vivo. CLI con entry points memory, memory-server, mcp-memory-server. No hay SDK JS oficial.

bModelo de datos

Clase Memory con identidad por content_hash externo (no UUID interno). Metadata extensible vía dict libre + propiedades SHODH (quality_score, credibility, emotion, episode_id, sequence_number). Ontología validada de 9-12 tipos base extensible por env var MCP_CUSTOM_MEMORY_TYPES.

cOperaciones de escritura · lectura

Embedding pre-computado NO se acepta en flujo normal. sqlite_vec.py:_generate_embedding(memory.content) siempre regenera. Solo MCP_EXTERNAL_EMBEDDING_URL permite usar Ollama/vLLM/TEI — pero el servidor llama, no recibe vector ya hecho. Cuello de botella para arquitecturas federadas.

Lectura: semantic (retrieve), hybrid BM25+vector (RRF), tag-based, temporal NL (memory_recall parsea expresiones en lenguaje natural), por tipo, graph traversal (memory_graph).

dConcurrencia · upstream

WAL mode + doble lock (asyncio + threading). Last-write-wins por content_hash. SHODH UMAS v1.0.0 mencionado en README pero ⚠ no se encuentra la spec ni en docs/ ni en wiki — posible vapor.

Bus factor 1. doobidoo con ~2298 commits; siguiente top contributor con ~36. Últimos 3 meses ~6 logins únicos (incluyendo dependabot). Esencialmente single-maintainer aunque con tracción y cadencia altísima (27+ releases en mayo 2026 solo).

Ciudadano técnicamente correcto pero territorialmente expansivo. La ABC MemoryStorage (11 métodos) es superficie limpia, pero el sistema asume que él computa embedding, gestiona consolidación, sirve dashboard, define ontología y es broker MCP. Sin webhooks salientes ni MCP notifications. Funciona como store-of-record único detrás de adaptadores, no como pieza igualitaria. El ritmo de releases + bus factor 1 → integrar vía REST/MCP estable mejor que forkear.

readOnlyHint/destructiveHint/idempotentHint aplicados a cada tool en server_impl.py. Permite policies de aprobación diferenciadas por destructividad. Replicar desde día uno.

Modelos pre-exportados en build, runtime sin torch. Reduce footprint un orden de magnitud sin sacrificar inferencia. Dos imágenes oficiales (full / cpu-quality) con misma API.

memory_recall (parser NL temporal) + memory_graph (transitive inference) + memory_consolidate (dream-inspired). Es la idea de "no solo CRUD vectorial" que diferencia el repo.

aSuperficie de integración

MCP server montado en /mcp/{bank_id}/ dentro del API server (HTTP). Binario aparte hindsight-local-mcp con stdio para Claude Desktop. 26 MCP tools single-bank, 29 multi-bank: retain, recall, reflect + mental models + directives + operations + tags. REST /v1/{tenant}/banks/{bank_id}/…. SDKs Python (hindsight-client) y Node/TS (@vectorize-io/hindsight-client). Helm chart producción. v0.6.0 añadió Oracle 23ai backend, v0.6.1 AlloyDB ScaNN.

bModelo de datos

Tablas Postgres documentadas: memories, entities, relations, temporal_facts, mental_models, banks, sessions, documents, opinions. Aislamiento multi-tenant a nivel de schema Postgres vía variable de contexto _current_schema. El paper organiza la memoria conceptualmente en cuatro redes lógicas: world facts, agent experiences, mental models (synthesized entity summaries) y opinions (evolving beliefs).

cOperaciones de escritura · lectura

Pipeline interno de 3 fases (DeepWiki): Extraction (LLM parsea contenido en atomic facts), Cognify (consolidación async por hindsight-worker), Indexing + Graph construction. Endpoint POST /consolidate para forzar cognify. ⚠ No hay endpoint para inyectar embedding precomputado documentado.

Lectura: 4 estrategias en paralelo (TEMPR) — Semantic (pgvector), Keyword (BM25), Graph (entity/temporal/causal), Temporal. Fusion por RRF + cross-encoder reranker. Budget levels low|mid|high modulan la activación. ⚠ Selector explícito por estrategia no documentado.

dConcurrencia · upstream

Postgres ACID. v0.5.5 introdujo per-operation slot reservations en worker pool. ⚠ Embedded pg0 probablemente no apto para múltiples APIs apuntando al mismo data dir.

Cadencia semanal confirmada (6 releases en 4 semanas). Sponsor: Vectorize AI Inc. Benchmarks en repo separado hindsight-benchmarks con leaderboard reproducible. LongMemEval: Gemini-3 91.4%, OSS-120B 89.0%, OSS-20B 83.6% vs 39.0% baseline (+44.6pp). Declara explícitamente que LoComo tiene "significant flaws" y recomienda LongMemEval.

Quiere ser el backend. Su valor de paper sale del stack TEMPR + RRF + cross-encoder + cognify worker actuando como sistema cerrado. Sí expone API limpios y multi-tenant nativo. Como caja-negra federable detrás de un router encaja bien. Como librería para arrancar componentes sueltos encaja mal — está diseñado como producto. No hay write-through trivial; si Naeth quiere ser dueño del schema o el flujo de scoring, Hindsight resiste.

aSuperficie de integración

MCP server (Docker HTTP o stdio local). Plugin Claude Code oficial con 5 hooks: SessionStart, PostToolUse, UserPromptSubmit, PreCompact, SessionEnd. REST API en :8000/docs. SDK Python (87.3% del codebase, patrón add → cognify → search async). CLI cognee-cli. SDKs extras: Vercel AI SDK adapter, Rust SDK (cognee-RS). Frameworks: LangGraph, Google ADK, OpenClaw, n8n, Claude Agent SDK, OpenAI Agent SDK. ⚠ Conflicto de puerto :8000 con mcp-memory-service.

bModelo de datos · ECL pipeline

Extract (add()): content-hash, metadata. Cognify (cognify()): 6 tasks — classify documents → permissions → extract chunks → extract graph (LLM) → summarize text → add data points. Load: persistencia al poly-store.

Poly-store: grafo (NetworkX/Neo4j/FalkorDB/Kuzu/Memgraph/Spanner/TuringDB) + vectorial (LanceDB/Qdrant/Pinecone/Weaviate/Redis/Milvus/OpenSearch/Valkey/PGVector/Azure) + relacional (SQLite/PostgreSQL/DuckDB).

Schema del KG emerge dinámicamente — sin ontología predefinida (4 prompts built-in + custom). Ontología opcional RDF/OWL externa vía RDFLibOntologyResolver. DataPoints Pydantic permiten ontología-like de facto.

cOperaciones de escritura · lectura

¿Saltar Cognify? Sí técnicamente — add() y cognify() son ops separadas. Pero si te saltas extract_graph y add_data_points no hay grafo ni vectores: solo metadata relacional. Aceptar memoria pre-procesada requiere bypaseo de adapters — posible pero no flujo de primera clase. Idempotencia: content-hash en add + pipeline-status en cognify.

Lectura: 15+ search modes (SearchType): GRAPH_COMPLETION (default), RAG_COMPLETION, CHUNKS/CHUNKS_LEXICAL, SUMMARIES, TRIPLET_COMPLETION, GRAPH_COMPLETION_COT, CYPHER, NATURAL_LANGUAGE, TEMPORAL, CODING_RULES, FEELING_LUCKY, FEEDBACK. Query a un solo store es posible — cada SearchType es elección del plano del poly-store.

dConcurrencia · upstream

Search en paralelo con cognify sin global lock. LanceDB tiene commit conflicts en multi-proceso. Transacciones distribuidas ⚠ no documentadas — eventual consistency entre 3 planos.

$7.5M seed funding. 80+ contributors. Adopters citados: Bayer (research workflows), University of Wyoming (evidence graph). Graduado del GitHub Secure OSS Program (ago 2025).

Se vende como framework pero internamente es library modular: pipeline ECL ensamblable, adapters de cada plano del poly-store sustituibles. Dicho esto, quiere ser el cerebro: el plugin de Claude Code engancha 5 hooks invasivos (UserPromptSubmit inyecta contexto, SessionEnd vuelca al grafo). Aceptarlo como "servicio especializado en KG generation" es viable consumiendo solo cognify() + SearchType.CYPHER/INSIGHTS sin adoptar el MCP server ni el plugin. Barrera real: 6 tasks de Cognify fuertemente acopladas al LLM.

★Ideas robables

cognee
#1
ECL como separation of concerns

Tres fases nítidas: Extract idempotente, Cognify costosa-LLM, Load multi-store. Optimizar cada una independientemente y skipear Cognify para datos ya estructurados.

cognee
#2
Poly-store con adapters comunitarios por DB

cognee-community aloja paquetes por DB. Patrón limpio para no casarse con un vendor manteniendo API uniforme.

cognee
#3
Plugin Claude Code con los 5 hooks de session lifecycle

SessionStart/UserPromptSubmit/PostToolUse/PreCompact/SessionEnd. Cualquier sistema de memoria persistente para Claude Code debería al menos contemplar PreCompact y SessionEnd.

cognee
#4
Search-type catálogo explícito + auto-routing

15+ modos nombrados con FEELING_LUCKY como auto-router. Mejor UX que un endpoint search() opaco. NodeSets como tagging ligero ortogonal a Datasets.

cognee
#5
Ontología opcional RDF/OWL externa sobre KG emergente

Mezclar schema emergente LLM con vocabulario canónico cuando el dominio lo requiere. Patrón híbrido valioso.

cognee
#6
Idempotencia content-hash + pipeline-status doble layer

Barato, robusto, no requiere transacciones distribuidas. incremental_loading=True por defecto.

cOperaciones de escritura · lectura

Ingesta de Markdown externo es caso de uso de primera clase. watch_service.py usa watchfiles.awatch y procesa eventos Added/Modified/Deleted sobre filesystem. Cualquier proceso externo puede dejar caer Markdown y Basic Memory lo recoge. Detección de moves por checksum entre deleted y added en el mismo batch. Atomic writes (write-then-rename). No hay file locks explícitos.

Búsqueda: filtros por note_types, entity_types, after_date, metadata_filters, tags, status, min_similarity. Paginación. Multi-proyecto con search_all_projects. Wikilink graph traversal vía build_context.

dImport/export · concurrencia · upstream

Export trivial (cp -r ~/basic-memory ./backup). Import nativo: ChatGPT, Claude conversations, Claude Projects, memory_json (Anthropic Memory MCP). Compatibilidad de facto con Obsidian/Logseq/Foam.

Filesystem-based: si humano edita en Obsidian mientras Claude escribe vía MCP, last-write-wins a nivel de archivo, sin merge automático. v0.21.0 menciona "concurrent delete fixes" y "N+1 query eliminations" — upstream endureciendo concurrencia.

~79 releases totales, ~1329 commits en main. Org Basic Machines mantiene basic-memory-skills, basic-memory-benchmarks, port TS MIT.

El mejor ciudadano del top 5 para arquitectura compuesta. Su storage es Markdown en disco; cualquier sistema externo escribe/lee los mismos archivos sin pedir permiso. Triple superficie (MCP + REST + CLI) sobre el mismo modelo, tool annotations bien declaradas, atomic writes respetados. Como sistema principal también funciona (búsqueda híbrida con fastembed/sqlite-vec, grafo emergente vía wikilinks), pero techo de rendimiento depende del filesystem. Único bemol serio: AGPL-3.0 impone obligaciones si Naeth se hostea como SaaS y toca código de Basic Memory en runtime.

aSuperficie de integración

MCP server propio en monorepo (mcp_server/ con releases independientes). Tools: add_episode, delete_episode, get_episodes, search_nodes, search_facts, delete_entity_edge, get_entity_edge, clear_graph, get_status. REST FastAPI en server/. SDK Python graphiti-core. Hybrid search con 3 rerankers seleccionables: RRF, MMR, cross-encoder (LLM o BGE).

bModelo de datos · bi-temporalidad

Tres primitivas: Episodes (dato crudo + provenance), Entities (nodes) con summaries evolutivos, Edges con validity window valid_at/invalid_at. Los facts no se borran al cambiar — se marca invalid_at. Custom Types vía clases Pydantic (ontologías prescritas) o emergentes (learned). add_episode acepta entity_types, excluded_entity_types, edge_types, edge_type_map.

cOperaciones de escritura · lectura

Pipeline canónico de add_episode: ingest → extract entities (LLM) → resolve duplicates (LLM + embedding match) → create/update edges con lógica temporal → invalidar edges contradictorios. add_episode_bulk sacrifica edge invalidation por throughput. LLM en path de escritura obligatorio; SEMAPHORE_LIMIT=10 controla concurrencia.

Lectura: hybrid search high-level graphiti.search(query, center_node_uuid, num_results) o low-level _search con SearchConfig. Temporal queries vía valid_at/invalid_at en resultados + reference_time. No hay DSL "AS OF timestamp" público — se resuelve filtrando por timestamps en cliente o vía Cypher directo.

dConcurrencia · upstream

ACID delegado al backend. Race conditions en entity dedup ⚠ — el pipeline depende del LLM para emparejar entidades; dos add_episode paralelos con la misma entidad nueva pueden producir duplicados antes de que la resolución alcance consistencia. group_id ("Graph Namespacing") es ciudadano de primera para multi-tenancy.

~2-4 releases/mes; 194 releases totales. Zep Community Edition fue deprecada en abril 2025; Graphiti absorbió la atención: alta cadencia, integración con Neptune, MCP con releases propias. Paper Rasmussen et al. arXiv:2501.13956 (DMR 94.8% vs MemGPT 93.4%; LongMemEval +18.5% accuracy con -90% latencia).

Integrable como capa de razonamiento temporal delegada, no como storage principal: enviar episodes y consultar hybrid search con metadata temporal, manteniendo Naeth como propietaria del corpus. Sin embargo Graphiti quiere ser fuente de verdad — su modelo asume ingesta de episodes crudos. El requisito de Neo4j/FalkorDB/Kuzu es fricción real para uso personal, aunque Kuzu (embedded) la reduce. Pega real: coste LLM por add_episode (extract + resolve + rerank). Buen ciudadano si se le delega solo el subdominio temporal/relacional.

Diagrama de puertos default #

Blueprint de los 5 sistemas con sus puertos por defecto y storage. Las líneas rojas punteadas marcan los conflictos detectados.

texto alternativo del diagrama
  • mcp-memory-service · SQLite-vec · :8000 REST + :8765 SSE
  • Hindsight · Postgres pg0 embedded · :8888 REST+MCP + :9999 UI
  • Cognee · poly-store · :8000 MCP HTTP — conflicto :8000 con mcp-memory-service
  • Basic Memory · filesystem .md · MCP stdio (sin puerto fijo)
  • Graphiti · Neo4j/FalkorDB/Kuzu · :8000 MCP HTTP + :7687 Neo4j — posible conflicto Neo4j con Cognee
  • Patrón propuesto Naeth: router :7777 enruta a los 5 backends, unifica auth + IDs + entity resolution.
conclusión

No hay incompatibilidades duras entre los 5 sistemas. Los únicos conflictos son resolubles por configuración (puerto :8000 colisión entre mcp-memory-service y Cognee; Neo4j compartido entre Cognee y Graphiti aislable por database). Y no hay conflictos de IDs entre sistemas — cada uno usa su propio espacio (content_hash, memory_id, UUID, paths FS). Si Naeth quiere un ID canónico unificado debe definirlo aparte y mapear.

§04hallazgos transversales

Patrones repetidos vs huecos de mercado #

Patrones repetidos: candidatos a primitivas de Naeth — si todos lo hacen, no inventes diferente. Huecos: ningún sistema lo tiene — donde Naeth puede aportar.

⊕ patrones repetidos (primitivas)

  • ⊕
    Content-hash como ID natural. Ningún sistema confía solo en autoincrement. Naeth debe seguir el patrón.
  • ⊕
    Variante de entity/observation/relation. Explícita en MM-S y Basic Memory; emergente en Cognee y Graphiti vía LLM extraction; implícita en Hindsight. Primitiva UMAS v2 candidata: triplets con metadata temporal opcional.
  • ⊕
    Hybrid search semantic + BM25. Variaciones del mismo patrón (TEMPR, tri-way, retrieve_hybrid, SearchRetrievalMode.HYBRID, GRAPH_COMPLETION).
  • ⊕
    MCP + REST + SDK + CLI — tetralogía de superficies estable en los 5. Planificar las cuatro desde día uno.
  • ⊕
    Embeddings 100% local viable en los 5 (sentence-transformers / ONNX / fastembed / Ollama). Cero excusa para depender de API.
  • ⊕
    Tool annotations FastMCP en MM-S y Basic Memory; soportadas por estándar MCP en los demás. Naeth las debe usar sistemáticamente.
  • ⊕
    Multi-tenant nativo en los 5 (con nombres distintos: X-Agent-ID, user_id, NodeSets, workspaces, group_id).

⊖ huecos de mercado (espacio para naeth)

  • ⊖
    Webhooks salientes / MCP notifications al escribir. Solo SSE de MM-S dashboard y file watcher de Basic Memory. Ninguno tiene HTTP saliente estandarizado. Si Naeth quiere orquestar, debe diseñar event bus propio.
  • ⊖
    Embedding pre-computado externo como input estándar. Federar 3 sistemas implica re-embeddings triples. Naeth debe manejar cache externo + endpoints raw.
  • ⊖
    Cross-system identity / entity resolution. Si "persona Eneko" aparece en MM-S, Hindsight y Basic Memory, ninguno sabe que son la misma entidad. Primitiva clave de Naeth.
  • ⊖
    Export común reversible (roundtrip lossless). Solo Basic Memory es trivialmente legible. SHODH UMAS v2 podría definir este contrato.
  • ⊖
    Audit log estilo OTEL GenAI Semantic Conventions. Solo Ogham-MCP (top 15, no top 5) lo hace.
  • ⊖
    Cliente identidad como dimensión primaria. Todos tienen algo (agent_id, user_id, group_id…), pero ninguno expone claude.ai vs claude-code vs cursor vs humano-Obsidian como dimensión de query.

Inconsistencias observadas #

⚠ docs vs código

mcp-memory-service: README dice "24 tools" pero el código registra ~18 modernas + aliases legacy en compat.py. Lee con cuidado.

mcp-memory-service: README declara "SHODH Unified Memory API v1.0.0 compatible" pero la spec no se encuentra ni en docs/, wiki, integrations.md ni memory-ontology.md. Posible vapor.

Hindsight: README principal MIT pero sub-paquete hindsight-api aparece como Apache 2.0 en algún view. Auditar archivo a archivo antes de derivar.

Basic Memory: año en releases reportado como 2024 en GitHub UI pero v0.21.1 declarada como mayo 2026 en README. Discrepancia de timeline.

Riesgos transversales #

riesgo

Bus factor: mcp-memory-service depende esencialmente de doobidoo. En arquitectura compuesta, no apostar el camino crítico a su upstream.

Cadencia altísima de releases (MM-S cada 3 días, Hindsight semanal, Graphiti 2-4/mes) → re-validación constante de adapters. Pinning estricto, versionado de integraciones.

Comercial backers: Vectorize/Hindsight, Zep/Graphiti, Cognee con $7.5M seed, Basic Machines con Cloud paid tier. El roadmap OSS no es libre — ya pasó con Mem0/OpenMemory (sunset) y Zep CE (deprecada).

Licencia AGPL (Basic Memory): impone obligaciones si Naeth se hostea como SaaS y toca su código en runtime. Apache/MIT en los otros 4 son neutrales.

§00reconfiguración

Decisiones que ajustan el Paso 2 #

El Paso 2 eligió la espina dorsal de B con Graphiti+Kuzu como enrichment. Cuatro decisiones de sesión lo reconfiguran.

1 · Hardware del VPS

GTX 1660 (4 GB VRAM), Ryzen 5 5500, 64 GB RAM. 4 GB descartan LLM local de calidad (solo un 3-4B Q4, extracción pobre).

2 · Sin dependencia de LLM

Temporal ligero local-puro. Se cae Graphiti (LLM-heavy en escritura). Naeth queda LLM-light: cero LLM, cero API, cero coste.

3 · La composición se mantiene

Cambia el inquilino del enrichment: mcp-memory-service (LLM-light) aporta hybrid RRF, dedup semántico y parser NL temporal, con embeddings locales.

4 · Forma = B · cascada

Source of truth único = los .md de Basic Memory; mcp-memory-service es índice derivado regenerable.

nota

Al ser ahora dos sistemas LLM-light locales, la Arquitectura A (federación) vuelve a ser competitiva (sus contras en el Paso 2 venían del coste LLM-heavy). El spike mide B; los datos permitirán decidir si A merece otra prueba.

método

El proxy se implementa como harness Python in-process: no hace falta el transporte MCP real para medir la latencia de composición. El overhead MCP se anota aparte, pendiente del montaje en el VPS.

§05lo que no cubre

Fronteras (intencional) #

  • Transporte MCP real (claude.ai / Claude Code conectados) — se mide composición, no transporte.
  • UMAS v2 / contrato canónico — es Paso 4+, si la composición se valida.
  • Concurrencia humano-Obsidian + Naeth sobre los mismos .md — se valida en el VPS con uso real.
  • Bi-temporalidad de entidades — descartada con Graphiti; el temporal aquí es por frontmatter + parser NL, sin grafo.

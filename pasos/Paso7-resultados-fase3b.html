Un Authorization Server propio, de verdad

La Fase 3b convierte el OAuth validado en 3a en un AS real: NaethOAuthProvider persiste clientes, códigos y tokens en Postgres, y el authorize ya no auto-aprueba — redirige a un login de 1 usuario. El agujero de 3a (cualquiera obtenía token) queda cerrado antes de exponer.

§00qué se construyó

Persistencia + login #

Subclase NaethOAuthProvider(OAuthProvider) en lugar del InMemoryOAuthProvider de 3a. FastMCP sigue aportando el plumbing; aquí van las dos piezas de producción.

  • Persistencia Postgres: 4 tablas locales (no se sincronizan, como job): oauth_client (DCR), oauth_pending (puente authorize→login), oauth_code (códigos de un uso), oauth_token (access/refresh con paired_token para rotación).
  • Login de 1 usuario: authorize() guarda un pending y redirige a /login; el formulario valida usuario+contraseña (compare_digest) y solo entonces emite el code.

§02implementación

Decisiones #

  • Tokens opacos (nae_at_…/nae_rt_…) validados contra oauth_token, no JWT: para 1 usuario es más simple y el revoke/rotación es directo. (FastMCP trae JWTIssuer si en finally se quisiera JWT.)
  • Rotación de refresh: al refrescar se revoca el par anterior vía paired_token; reusar el refresh viejo da 400.
  • Conmutable: OAUTH_PROVIDER=postgres (3b) | memory (3a). OAUTH_ENABLED gobierna on/off, off por defecto (Claude Code sigue por localhost).
  • DB async-safe: las queries síncronas corren en threadpool (anyio.to_thread) para no bloquear el event loop. Tráfico OAuth de 1 usuario es esporádico.
§03caveats honestos

Lo que 3b NO cierra #

  • Sin sesión persistente: cada authorize pide login. Más seguro; para claude.ai apenas molesta (re-autoriza con el refresh token).
  • Sin CSRF ni rate-limit en /login: superficie mínima de 1 usuario; se endurece en finally si hace falta.
  • Contraseña en .env en claro: coherente con el resto de secretos del nodo. Pon una real antes de exponer.
  • No probado aún con claude.ai real: 3b valida con cliente OAuth propio + login por POST. El flujo real (DCR desde la nube + redirect de navegador + tu login en pantalla) se prueba en la Fase 4.
veredicto fase 3

Naeth tiene un Authorization Server OAuth 2.1 propio, persistido en Postgres y con login de 1 usuario: DCR, authorization code + PKCE, refresh con rotación, revocación, y el discovery + 401 que claude.ai exige. El gate de login cierra el riesgo de auto-aprobación. La Fase 3 está completa. Procede la Fase 4: túnel a naeth-local.enraxk.dev, OAUTH_BASE_URL público y alta del connector en claude.ai.

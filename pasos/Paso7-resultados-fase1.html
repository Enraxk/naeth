§00pila levantada

Qué se construyó #

La pila base del Paso 7: db (pgvector pg17) + api (FastAPI: visor + CRUD, sin MCP aún) + worker (embeddings async), en loopback sobre Docker Desktop. El núcleo del Paso 6 aplicado tal cual: 6 tablas, la vista memory_current y los índices (HNSW parcial sobre is_current, GIN sobre tsv/tags). Dimensión vector(384) renderizada desde EMBED_DIM.

Build del índice: 0.14 s. Recall 0.96 a latencia sub-ms: los defaults de pgvector valen para Naeth; no hace falta subir ef_search. El caveat del Paso 6 §8 (el spike midió latencia con vectores sintéticos, no recall real) queda cerrado.

§03ajuste forzado por la librería

e5-small → MiniLM multilingüe #

modelo

El Paso 7 §8 fijaba e5-small (384) para el nodo local, pero fastembed 0.5.1 no lo incluye (solo multilingual-e5-large, 1024). Se usó el equivalente multilingüe soportado: paraphrase-multilingual-MiniLM-L12-v2 (384-dim, ligero, español). Mismo tamaño de vector → no toca el schema; reversible por env var. bge-m3 / e5 entran en la Fase 2, que es donde el plan los sitúa.

§04caveats honestos

Lo que estos números NO dicen #

  • Recall plano (0.96 en todos los ef_search): el corpus es generado por plantillas (baja diversidad, near-duplicates), así que el índice acierta casi todo ya con ef=10. Una curva recall-vs-ef_search representativa pide un corpus real variado; es refinamiento de medición, no bloquea la Fase 1.
  • Sin MCP/OAuth/túnel todavía: la Fase 1 es solo pila base + visor local. MCP (Fase 2), OAuth 2.1 (Fase 3) y túnel enraxk + claude.ai (Fase 4) vienen después.
  • pgdata aún en C:: arrancó en el volumen Docker por defecto. Mover la disk image a F: (Settings → Resources) queda pendiente; con datos mínimos es trivial.
  • No es calidad de recall en español: recall@k mide HNSW vs exacto (¿el índice recupera lo que la fuerza bruta?), no si el modelo entiende bien el español. Eso es la Fase 2.

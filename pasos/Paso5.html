El sustrato: Postgres + pgvector

Dos diseños de Naeth v1 en conflicto (Postgres propio vs componer Basic Memory). Criterio rey: que el grafo de nodos y la interfaz escalen a la perfección. El eje que decide es la búsqueda semántica a escala, y ahí pgvector con HNSW gana de forma decisiva, verificado a 1M de notas.

§01el eje decisivo

Búsqueda semántica a escala #

El grafo vive de las aristas semánticas (kNN sobre embeddings). Es la operación más frecuente y la que más crece. Aquí los dos no juegan en la misma liga.

verificado

Medido el 2026-05-30 en el server finally (Ryzen 5 5600X, CPU-only). A 1M, pgvector es ~100× más rápido y casi no sube. Nota de despliegue: construir el índice HNSW de 1M tardó 5,7 min con tuning (maintenance_work_mem=2GB, max_parallel_maintenance_workers=4, Docker --shm-size=8g); por defecto serían ~40 min.

caveat

El spike midió latencia con vectores sintéticos, no recall real ni el eje ef_search latencia-vs-recall. Se verifica con embeddings reales cuando haya datos. No bloquea la decisión.

§02aristas y recorrido

El grafo, en cada sustrato #

  • Postgres: relaciones en una tabla; vecindad a N saltos con WITH RECURSIVE. Combinas aristas + filtros (SQL) + kNN (vector) + texto (tsvector) en una query.
  • Basic Memory: relaciones de wikilinks vía build_context (forma limitada); para un grafo a medida acabas leyendo su SQLite directo, acoplándote a un esquema que cambia cada release (AGPL, single-maintainer).
§03la interfaz

Control total de las queries #

En Postgres escribes exactamente las queries que el grafo y la UI necesitan (vecindad, kNN con umbral, agregados, filtros) y reusa tu patrón MCP (X-Enraxk-Token). Basic Memory te encierra en su modelo o te obliga a hackear su DB.

§04basic memory revisado

Su baza se cae al quitar Obsidian #

  • Soberanía como Markdown legible era su gran ventaja, pero descartas Obsidian al 100%: ese beneficio se desvanece. Y no la pierdes con Postgres (export a Markdown o pg_dump).
  • Menos código al inicio: cierto y honesto. A cambio heredas su modelo, su licencia y su ritmo.
  • El spike no se tira: validó principios (recall barato, el coste es el embedding, cola async) agnósticos del sustrato. Tu diseño claude.ai ya pedía esa cola async.

decisión

Cumple tu prioridad uno (escalar grafo + UI), reusa tu patrón, ADD-only nativo, y reconcilia con tu diseño canónico de claude.ai. Basic Memory era buena para "memoria personal con Markdown + Obsidian"; al cambiar la meta a "grafo escalable + interfaz propia", su punto fuerte cae.

§06qué cambia

Si se elige Postgres #

  • El Paso 4 se mantiene casi entero (heptápodo, grafo global+vecindad, embeddings, adjuntos, Cloudflare, LUKS, Milkdown). Cambia el modelo de datos: nodos = filas, no ficheros; el "árbol" pasa a ser jerarquía lógica (path).
  • El visor del spike (sobre Basic Memory) queda como prototipo desechable; la v1 de la web se hace sobre Postgres.
  • El esquema concreto está en Paso 6 (paso6-esquema.md).

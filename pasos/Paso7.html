Naeth corriendo en este equipo

Levantar Naeth v1 primero en este Windows sobre Docker (Postgres + pgvector), con su servidor MCP para Claude Code (local) y claude.ai (por túnel), antes de migrar a finally. claude.ai entra con OAuth 2.1 nativo en Naeth; los embeddings son por-nodo; el local nace como nodo multi-master.

§01por qué local primero

Local-first, por los apagones #

El deploy directo a finally (hoy Paso 9) tiene un problema operativo real.

  • Naeth corre primero en este equipo: se prueba y testea hasta que funcione como se quiere, y solo entonces se migra.
  • El destino es multi-master: este equipo y finally como nodos que reconcilian (Paso 8).
  • Por eso el local nace con Postgres propio completo, no como cliente del VPS.
el motivo

Los apagones tumban finally y hay que esperar a que vuelva. Naeth debe seguir vivo y escribible en local durante la caída, y reconciliar cuando el server vuelva. Ese requisito es lo que define toda la V1.

El núcleo es el del Paso 6 (memory ADD-only, relation, attachment, job; búsqueda híbrida RRF). §04red local

Quién llega por dónde #

Claude Code (este equipo) ──localhost──▶ 127.0.0.1:PUERTO_MCP   (bearer en header)

navegador (este equipo) ──localhost──▶ 127.0.0.1:PUERTO_WEB   (visor; sin túnel)

claude.ai (nube) ──HTTPS──▶ Cloudflare edge ──túnel enraxk──▶ 127.0.0.1:PUERTO_MCP
                                                            (OAuth 2.1 lo hace Naeth)
  • Claude Code ↔ MCP: directo por localhost con --transport http --header "Authorization: Bearer …". Sin OAuth.
  • claude.ai ↔ MCP: es cloud, no alcanza localhost → necesita HTTPS público → túnel enraxk. No admite bearer estático, solo OAuth.
  • Visor web: solo localhost, no se expone por el túnel.
  • Hostname: naeth-local.enraxk.dev (no pisa el naeth.enraxk.dev del VPS).
§05servidor MCP

Las herramientas v1 #

Transport Streamable HTTP (SSE es el legado). Mismas herramientas para claude.ai y Claude Code; el visor es gestión.

  • memory.add — alta de memoria (síncrona) + encola embedding (job).
  • memory.search — búsqueda híbrida RRF (semántica + léxica).
  • memory.get — recupera una memoria y su cadena de versiones.
  • memory.supersede — nueva versión (ADD-only) que reemplaza a otra.
  • memory.tombstone — marca de borrado lógico.
  • relation.add / relation.list — grafo.
  • system.status — salud (cola de embeddings, conteos, modelo activo).
§06OAuth 2.1 (claude.ai)

Lo crítico: la auth la hace Naeth #

claude.ai exige OAuth 2.1 con PKCE; no admite OAuth 2.0 plano ni bearer estático. Naeth actúa como servidor de autorización + de recurso.

discoveryPublicar /.well-known/oauth-protected-resource y /.well-known/oauth-authorization-server (RFC 9728 / spec MCP).
401Sin token, responder 401 con WWW-Authenticate: Bearer resource_metadata="…". Esto es lo que rompía con Cloudflare Access.
flujoAuthorization Code + PKCE (code_challenge/code_verifier), access token + refresh token y endpoint de refresh.
registroDynamic Client Registration (DCR): claude.ai registra su cliente y aporta su redirect_uri → no hay que hardcodear Client ID/Secret ni callbacks.
identidadMemoria personal de un usuario → tokens para un único principal. Sin multi-tenant. Se reusa en finally y en el sync.
por qué nativo gana

El bug anthropics/claude-ai-mcp #410 (cerrado not-planned) demostró que claude.ai web/móvil exige el header WWW-Authenticate en el 401 y abandona si falta; Claude Code lo tolera. Cloudflare Access Managed OAuth no lo emitía → poner Access delante del MCP rompe claude.ai. Al hacer el OAuth nativo en Naeth, el header se emite siempre y claude.ai funciona — además de no atar la auth a ningún vendor.

§07docker + túnel

Empaquetado y exposición #

db     pgvector/pgvector · volumen Docker nombrado (NO bind-mount NTFS) · solo loopback
api    FastAPI/uvicorn · MCP + OAuth + visor · bind 127.0.0.1:PUERTO
worker embeddings en CPU · consume la cola job

cloudflared: naeth-local.enraxk.dev ──▶ 127.0.0.1:PUERTO_MCP   (solo el MCP)
  • Discos (2026-06-24): pgdata → F: (SSD SATA; se mueve la disk image de Docker Desktop a F:, sigue siendo volumen gestionado/ext4, no bind-mount). Binarios → E: (HDD 1.8 TB; E:\naeth\assets como bind-mount — NTFS vale para servir blobs). C: (sistema) se deja libre.
  • pgdata en volumen Docker (backend WSL2), nunca bind-mount a ruta NTFS: Postgres rinde mal sobre el FS montado de Windows.
  • Solo el endpoint MCP va por el túnel; visor e inspección de DB se quedan en localhost.
  • En local no aplica el endurecimiento de finally (UFW, fail2ban, LUKS); la única superficie pública es el MCP, protegido por el OAuth de Naeth.
nada de Access aquí

Sin Cloudflare Access delante del MCP — eso es justo lo que rompe claude.ai (§06). Si algún día se quisiera doble capa, la vía compatible sería el MCP Server Portal de Cloudflare, no Access clásico. En v1 no se usa.

§08embeddings por-nodo

Cada nodo, su modelo #

Como cada nodo regenera sus embeddings y la cola job no se sincroniza, el modelo es una decisión por-nodo, no global.

nodomodelodimpor qué
este equipobge-m31024evaluar calidad del recall en español; el equipo da para ello
finally (VPS)multilingual-e5-small384preparado para sus recursos; mantiene el techo ~100/s del spike
  • Cada nodo debe ser internamente consistente (mismo modelo para indexar notas y para embeber la query).
  • El sync transfiere solo filas (memory/relation/attachment, ADD-only, id uuid); nunca embeddings → la divergencia de modelo/dimensión entre nodos es inofensiva.
  • La dimensión vector(N) y el modelo son configurables por env var; si bge-m3 no compensa en local, e5-small es el fallback inmediato.
§09stack y fases

Stack y ejecución #

Stack: FastAPI + SDK MCP/FastMCP + Authlib + pgvector + fastembed/ONNX. Se ejecuta por fases, cada una con OK.

1 · pila basecompose db + api + worker; migraciones del esquema con dimensión parametrizada; visor en localhost. Verificar HNSW aquí (m=16, ef_construction=64 y medir).
2 · MCP localexponer las herramientas MCP; validar con Claude Code por localhost. Aquí se prueba la memoria de punta a punta y se evalúa bge-m3 en español.
3 · OAuth 2.1discovery + authorize + token + refresh + PKCE + DCR + 401 con WWW-Authenticate.
4 · túnel + claude.aicloudflared a naeth-local.enraxk.dev; alta del custom connector en claude.ai; verificar el flujo OAuth completo.

§00tl;dr · decisión

Tres topologías, una apuesta #

Cada arquitectura responde distinto a dónde vive la verdad y quién paga el coste. Ninguna pura gana en abstracto; la que de-riesga primero lo que no sabemos es B.

A · federación lateral[router]

Naeth = scatter-gather

Backends como pares; escritura fan-out, lectura en paralelo con merge + RRF. Máxima cobertura, pero agrava casi todas las preguntas duras: latencia = el más lento, N embeddings, consistencia de escritura sin reconciliador, identidad cross-system a cargo de Naeth.

B · pipeline en cascada[spike]

Hot tier + enrichment async

Hot tier síncrono (Basic Memory, Markdown soberano) sirve el recall rápido; un enrichment async añade grafo/temporal por detrás vía bus propio de Naeth. Acota la latencia (pregunta 1) y fija un source of truth único (preguntas 4 y 8).

C · spec + adaptadores[después]

UMAS v2 + driver ABC

Naeth define el contrato de memoria; cada backend vive tras un adaptador conmutable. Da portabilidad real y roundtrip lossless (el hueco que nadie cubre), pero a costa de la mayor ingeniería y del riesgo de aplanar el valor de los backends ricos.

decisión

Spikear primero la espina dorsal de B con el subconjunto mínimo, no los 5 sistemas. Naeth como 1 MCP proxy → Basic Memory como hot tier (AGPL aislada por la frontera "solo leo/escribo sus .md") → un enrichment async (Graphiti-Kuzu para temporal, o Cognee para KG; no ambos). El destino probable es un híbrido B+C, con A como modo de lectura opcional.

por qué

La pregunta 1 (¿latencia <2s al componer memoria?) es la única que el Paso 1 marcó como "solo medible en spike", y las tres arquitecturas dependen de su respuesta. B la pone en el centro del spike sin contaminar la medición con un backend LLM-heavy en el camino crítico (eso haría A), y sin gastar antes la ingeniería del contrato UMAS v2 (eso es C, que se diseña después de confirmar viabilidad).

Naeth es un router sin storage propio. Los backends son pares; cada uno es dueño de su porción. La escritura hace fan-out; la lectura hace scatter-gather en paralelo y Naeth fusiona con RRF (idea robable de Hindsight) y trunca. Un solo MCP server proxy evita exponer las 91 tools.

                          ┌──────────────────────────────┐
   claude.ai  ──MCP──▶    │            NAETH             │
   Claude Code ─MCP──▶    │  (1 MCP server proxy)        │
                          │  write fan-out / read merge  │
                          └───┬───────┬───────┬───────┬──┘
                   write+read │       │       │       │
                 ┌───────────┘   ┌───┘   ┌───┘   ┌───┘
                 ▼               ▼       ▼       ▼
             MM-S            Hindsight  Cognee  Basic Memory  (Graphiti…)
          (rápido)         (TEMPR)    (KG)    (Markdown)

▲ pros

  • +Aprovecha lo mejor de cada backend a la vez sin tener que elegir.
  • +Degradación graceful natural: cae un backend, hay resultado parcial (pregunta 3).
  • +Sin lock-in; cada backend reemplazable en caliente.

▼ contras

  • !Latencia = el más lento + el reranker (pregunta 1). Un LLM-heavy en lectura dispara el p95.
  • !Escritura sin reconciliador: fan-out falla en 1 de N → estado parcial (pregunta 4).
  • !N embeddings + N facturas LLM por escritura (preguntas 5 y 10).
  • !Identidad cross-system y respuesta >1MB MCP quedan a cargo de Naeth (preguntas 6 y 2).

Los backends se ordenan por velocidad y coste, no como pares. Un hot tier síncrono y LLM-light absorbe la escritura y sirve el recall rápido. Un enrichment tier LLM-heavy consume del hot tier en segundo plano y añade grafo/temporal/consolidación. Naeth corre el bus que los conecta, porque ningún backend emite eventos (hueco 4.2.1 del Paso 1).

                      ┌───────────────────────────────────────┐
  claude.ai ─MCP─▶    │                 NAETH                 │
  Claude Code─MCP▶    │  1 MCP proxy · write síncrono al hot  │
                      └───┬─────────────────────────────▲─────┘
            write+recall │                             │ recall escalado
                         ▼                             │ (si hot no basta)
                 ┌───────────────┐                     │
                 │  HOT TIER     │ ── source of truth ─┘
                 │  Basic Memory │   (Markdown en disco, soberano)
                 │  o MM-S       │
                 └──────┬────────┘
                        │ event bus de Naeth (async, propio)
                        ▼
                 ┌───────────────────────────────┐
                 │  ENRICHMENT TIER (async)       │
                 │  Graphiti (temporal)  ó        │  ← uno, no ambos (Q9)
                 │  Cognee (KG)          ó        │
                 │  Hindsight (TEMPR)             │
                 └───────────────────────────────┘
                 proyección/índice derivado, regenerable

La verdad es el hot tier. Recomendación de papel: Basic Memory, cuyo storage es Markdown en disco (el "mejor ciudadano" del Paso 1) → legible, versionable, regenerable. El enrichment es proyección derivada: si se corrompe o el upstream pivota, se reconstruye releyendo los .md. ⚠ La AGPL de Basic Memory no se propaga si Naeth solo lee/escribe sus archivos sin enlazar su código en runtime — mantener esa frontera es decisión de diseño.

▲ pros

  • +Latencia de recall acotada: el camino crítico no toca LLM-heavy (pregunta 1).
  • +Source of truth único: partial write tolerable, sin saga ni rollback (preguntas 4 y 8).
  • +FIFO controlable en un único punto de entrada (pregunta 12).
  • +El event bus que obliga a construir es necesario en las tres arquitecturas → inversión reutilizable.
  • +Coste LLM y embeddings sobre un backend, diferidos fuera del camino crítico (preguntas 5 y 10).

▼ contras

  • !El enrichment va por detrás (eventual): un recall que necesite el grafo recién escrito puede no encontrarlo aún. ⚠ a validar en spike.
  • !Naeth debe construir y operar el bus + worker.
  • !Reprocesar al enrichment reintroduce coste LLM y embeddings (diferido y acotado a un backend).

Naeth define un contrato de memoria canónico (UMAS v2: triplet entity/predicate/value + metadata temporal + identidad de cliente) y cada backend vive tras un adaptador que implementa una ABC tipo MemoryStorage (idea robable de MM-S; el patrón factory.py ya existe upstream). UMAS v2 puede ser el roundtrip lossless que ningún sistema ofrece hoy (hueco 4.2.4).

   claude.ai ─MCP─▶ ┌─────────────────────────────────────────┐
   Claude Code─MCP▶ │                 NAETH                   │
                    │  UMAS v2 (contrato canónico de memoria) │
                    │  + storage canónico propio (opcional)   │
                    └──┬──────────┬──────────┬──────────┬─────┘
                       │ adapter  │ adapter  │ adapter  │ adapter
                       ▼          ▼          ▼          ▼
                     MM-S     Basic Mem.   Cognee    Graphiti
            (cada adaptador traduce UMAS v2 ⇄ modelo nativo del backend)

C1 · un backend activo conmutable (como MM-S factory.py): eliges uno; Naeth te da portabilidad para cambiar sin perder datos. C2 · storage canónico propio + write-through: Naeth posee la verdad y resuelve el embedding canónico (pregunta 5) computándolo una vez sobre su store.

▲ pros

  • +Portabilidad real: cambiar de backend sin perder datos (mitiga pivotadas comerciales y bus factor 1).
  • +Contrato de memoria explícito y versionado (pregunta 7); roundtrip lossless (hueco 4.2.4).
  • +1 MCP server limpio (pregunta 11); embedding canónico resuelto en C2 (pregunta 5).

▼ contras

  • !Definir y mantener UMAS v2 + N adaptadores persiguiendo upstreams que releasan cada 3 días a semanal.
  • !Los backends ricos resisten ser drivers: reducir Graphiti a put/get tira su bi-temporalidad; el mínimo común aplana el valor.
  • !Escribir embeddings canónicos a los backends exige bypassear sus APIs — frágil ante cada release. ⚠ a validar.

§03decisión · spike

Qué validar primero, y por qué #

No se construye la arquitectura final en el spike. Se valida la suposición que, si es falsa, invalida todo lo demás: la pregunta 1.

▶Spike: espina dorsal de B #

subconjunto mínimo
  • Naeth como 1 MCP server proxy (cierra la pregunta 11).
  • Hot tier = Basic Memory (Markdown como source of truth soberano; AGPL aislada por la frontera "solo leo/escribo sus .md"). Escritura síncrona.
  • Event bus propio mínimo de Naeth (el hueco 4.2.1; necesario en A, B y C).
  • Enrichment = exactamente uno, async: Graphiti con Kuzu embebido (la opción menos invasiva para uso personal) si se valida temporal, o Cognee si lo prioritario es KG. No ambos (pregunta 9).

Qué mide el spike #

Cada medición cierra una pregunta. Marca las que priorices para el Paso 3.

no A, no C

No A: pondría un backend LLM-heavy en el camino crítico de lectura desde el minuto cero, contaminando la medición de latencia con el peor caso antes de saber si el caso bueno es aceptable. No C: su mayor coste es definir UMAS v2 y los adaptadores; gastarlo antes de saber si componer es viable es invertir en el contrato de algo que quizá no compense. C se diseña después, y el hot tier de B puede adoptar UMAS v2 más tarde — B y C son componibles, no excluyentes.

destino

Si el spike sale bien: B-mínima → añadir UMAS v2 al hot tier (empieza C sobre B) → considerar federación lateral (A) solo para lecturas sobre backends ya poblados, nunca para escrituras. El destino probable es un híbrido B+C con A como modo de lectura opcional, no una de las tres puras.

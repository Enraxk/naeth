El núcleo de Naeth en Postgres

El modelo de datos que usan las dos interfaces (web y MCP) sobre el mismo núcleo. ADD-only append puro (reemplazo y borrado como tablas-evento), búsqueda híbrida local, adjuntos, cola async, y listo para sync multi-master por unión de filas (Paso 8). Embeddings por-nodo (Paso 7).

§01principios

Lo que fija el esquema #

  • ADD-only estricto: editar y borrar no pisan datos; añaden versiones y eventos. El reconcile por LLM destruye contexto, por eso se descarta.
  • Append puro para multi-master (rev. 2026-06-24): lo "mutable" (reemplazo, validez, borrado) se modela como filas-evento inmutables en tablas propias (supersession, tombstone) → sincronizar es unir filas en todas las tablas (Paso 8).
  • Lo derivado es caché local: is_current se recalcula en cada nodo desde las tablas-evento (para índices parciales). No viaja en el sync.
  • Búsqueda híbrida local: semántica (pgvector HNSW) + texto (tsvector) + filtros, en una query. Sin LLM, sin API.
  • Soberanía por export: la verdad vive en Postgres, portable (pg_dump, export a Markdown). Sin lock-in.
  • Identidad de cliente: cada escritura registra quién la hizo (claude.ai / claude-code / web / mcp).
  • Embedding asíncrono y por-nodo: la escritura es síncrona y barata; el embedding va a una cola. Modelo y dimensión son por-nodo y la columna no se sincroniza.
§02tabla memory · inmutable

qué cambió · rev 2026-06-24

Se quitaron de la fila supersedes, valid_to y deleted_at. El reemplazo vive en supersession (§03), el borrado en tombstone (§04), y la vigencia es is_current (caché). embedding pasa de vector(384) fijo a vector(N) por-nodo. La fila memory queda 100% inmutable salvo el relleno async del embedding y la recomputación de is_current (ambos locales).

crearINSERT en memory. Si el content_hash ya existe vigente, idempotente.
editarINSERT versión nueva + INSERT en supersession (child=nueva, parent=anterior). La anterior permanece; is_current=false.
borrarINSERT en tombstone (target=la fila). La fila permanece; is_current=false.
fundirINSERT versión fusión + supersession con una fila por rama padre. Resuelve el conflicto sin perder ninguna rama (Paso 8).

Vista de lo vigente (la verdad, derivada de las tablas-evento):

CREATE VIEW memory_current AS
SELECT m.* FROM memory m
WHERE NOT EXISTS (SELECT 1 FROM tombstone t WHERE t.target_id = m.id)
  AND NOT EXISTS (SELECT 1 FROM supersession s WHERE s.parent_id = m.id);
-- is_current cachea este predicado para los índices parciales (§08).
§03supersession

Versionado append-only #

El reemplazo entre versiones como aristas inmutables (antes la columna supersedes). Permite varios padres para fundir ramas.

id uuid PK · child_id uuid → memory · parent_id uuid → memory
created_at timestamptz · source_client text
  • Edición normal → una fila (child reemplaza a un parent).
  • Fusión → N filas con el mismo child y distintos parent (una por rama).
  • parent_id con >1 child vigente = conflicto de rama (dos ediciones offline); se detecta y se marca para fundir:
SELECT s.parent_id, array_agg(s.child_id) AS ramas
FROM supersession s JOIN memory_current c ON c.id = s.child_id
GROUP BY s.parent_id HAVING count(*) > 1;
§04tombstone

Borrados append-only y unificados #

El borrado lógico como evento inmutable (antes la columna deleted_at). Unificado para memory y relation.

id uuid PK · target_id uuid · target_kind text (memory | relation)
created_at timestamptz · source_client text
  • "Borrado" = existe una fila tombstone para ese target_id. Nunca hay DELETE físico ni UPDATE.
  • Sincroniza por unión; la unión de tombstones es idempotente (un borrado en cualquier nodo gana, monotónico).
§05relation (grafo)

Aristas explícitas #

id uuid PK · source_id uuid → memory · target_id uuid → memory
predicate text (links_to, depends_on, ...) · metadata jsonb
source_client text · created_at timestamptz
  • ADD-only: las relaciones no se borran in-place; se registran en tombstone (target_kind='relation'). Vigente = NOT EXISTS tombstone.
  • Recorrido de vecindad (foco + N saltos) con WITH RECURSIVE + índices en source/target.
  • Las aristas semánticas NO se almacenan: se calculan bajo demanda con pgvector kNN (2-4 ms a 1M). El grafo "qué se parece" pide las K vecinas del nodo en foco, sin materializar millones de aristas.
§06attachment (q6)

HTML, PDF, cualquier archivo #

id uuid PK · memory_id uuid → memory -- el sidecar que lo representa
filename · mime · size_bytes · sha256
storage_path text -- ruta en el volumen LUKS; los BYTES no van en Postgres
extracted_text text -- PDF/HTML → texto; alimenta content/tsv/embedding
  • El binario vive en disco (volumen LUKS), no en la BD (evita bloat).
  • Cada adjunto genera un sidecar memory (content = texto extraído) → entra en búsqueda, embedding y grafo como un nodo más.
  • Extracción: PDF con PyMuPDF, HTML con trafilatura, texto plano inline; imágenes → metadatos (OCR opcional).
  • Sync (Paso 8): la fila viaja por unión; el binario se replica content-addressed por sha256 (transferir faltantes, dedup).
§07job · cola local

tabla job (no se sincroniza) #

La async write queue del diseño, y lo que el visor muestra. Local a cada nodo.

id bigserial PK · kind text (embed | extract | ...) · memory_id uuid
status text (pending|processing|done|error) · attempts int
created_at · started_at · finished_at · error text
  • Tras un INSERT en memory se encola job(kind='embed'). Un worker lo toma, genera el embedding en CPU y hace UPDATE memory SET embedding (mutación local).
  • bigserial es local; la cola no se sincroniza: cada nodo regenera sus embeddings con su modelo (Paso 7).
  • Da observabilidad (el visor lee de aquí) y reintentos. Desfase = finished_at - created_at.
§08índices y tuning

Índices #

-- semántico: HNSW solo sobre lo vigente y embebido (usa la caché is_current)
CREATE INDEX ON memory USING hnsw (embedding vector_cosine_ops)
  WHERE embedding IS NOT NULL AND is_current;
CREATE INDEX ON memory USING gin (tsv);     -- texto completo
CREATE INDEX ON memory USING gin (tags);    -- etiquetas
CREATE INDEX ON memory (path);              -- árbol
CREATE INDEX ON supersession (parent_id);   -- vigencia + sync
CREATE INDEX ON tombstone (target_id);
CREATE INDEX ON relation (source_id);
CREATE INDEX ON job (status) WHERE status = 'pending';
por qué is_current existe

El índice parcial HNSW usa is_current (caché de la propia fila) porque un predicado de índice parcial no puede referenciar otras tablas. Por eso la vigencia se cachea en la fila (no es la verdad: se recomputa tras cada edición/borrado/sync desde supersession y tombstone).

tuning de build (Paso 5)

Solo afecta al BUILD del HNSW, no a las queries. maintenance_work_mem=2GB (el grafo en RAM → 7,4× build y 2,7× query) · max_parallel_maintenance_workers=4 · Docker --shm-size=8g (el build paralelo usa /dev/shm; 64 MB por defecto da un DiskFull engañoso — crítico en local con Docker Desktop y en finally). Ojo: con bge-m3 (1024-dim) el grafo ocupa ~2,7× más que con 384-dim → recalcular maintenance_work_mem en el nodo local.

§09búsqueda híbrida

Una sola query (RRF) #

WITH sem AS (
  SELECT id, row_number() OVER (ORDER BY embedding <=> :q) AS r
  FROM memory_current WHERE embedding IS NOT NULL
  ORDER BY embedding <=> :q LIMIT 50),
txt AS (
  SELECT id, row_number() OVER (ORDER BY ts_rank(tsv, plainto_tsquery(:kw)) DESC) AS r
  FROM memory_current WHERE tsv @@ plainto_tsquery(:kw) LIMIT 50)
SELECT m.*, (1.0/(60+sem.r) + 1.0/(60+txt.r)) AS score
FROM memory_current m
LEFT JOIN sem ON sem.id = m.id
LEFT JOIN txt ON txt.id = m.id
WHERE sem.id IS NOT NULL OR txt.id IS NOT NULL
ORDER BY score DESC LIMIT :k;

La query embebe :q con el modelo de ESTE nodo (consistencia interna por-nodo). Filtros estructurados (path, tags, fechas, source_client) se añaden con WHERE normal.

§10MCP

La otra interfaz, mismo núcleo #

El núcleo no sabe si lo llama la web o un agente: ambos pasan por la misma API (FastAPI) que impone ADD-only. Herramientas MCP v1 (Paso 7): memory.add, memory.search, memory.get, memory.supersede, memory.tombstone, relation.add, relation.list, system.status. No hay editar/borrar destructivo: editar = versión + supersession; borrar = tombstone. La política vive en el núcleo, no en cada cliente.

fin paso 6 (rev.)

El esquema nace listo para multi-master: ADD-only append puro (supersession + tombstone como tablas-evento), is_current como caché local, embeddings por-nodo. Sincronizar es unión de filas, sin reconciliar mutaciones. Lo que NO decide: el código (DDL, migraciones, API), el detalle de extracción por tipo de archivo, y particionado (innecesario a escala personal). Siguiente: construir, empezando por el Paso 7.
